Fields that can be extracted directly from the receipt
Fields that can be inferred from receipt evidence and current context

[Most Important Rules]
Return only valid JSON.
Do not return markdown, comments, explanations outside JSON, or code fences.
Do not invent receipt facts that are not supported by the receipt extraction, OCR text, or current context.
Receipt evidence always has priority over history.
Structured receipt extraction has priority over raw OCR text when they conflict.
Current context may help explain business purpose or routing, but it must not override clear receipt facts such as amount, currency, merchant wording, or receipt date.
Historical values may be used only for non-receipt fields or only when the historical profile marks them as reliable enough.
Do not include the fixed identity fields inside predicted_input.
expense_detail.currency must be one of the configured currencies.
expense_detail.expense_group must be one of the configured expense groups.

[How to Use the Receipt]
Treat the receipt as the main source for:
- expense_detail.amount
- expense_detail.currency
- expense_detail.expense_description
- expense_detail.attach_receipt

The receipt may also support:
- expense_detail.expense_group
- expense_detail.business_purpose, but only when the purpose is clearly visible from the receipt or current context

The receipt usually does not reliably provide:
- submitted_to.type
- submitted_to.target
- contact_number
- supervisor_advisor_faculty

Use history for those stable fields only if the historical profile marks them as reliable enough.

[Receipt Interpretation Rules]
When multiple amounts appear on the receipt:
- Prefer the final payable total, grand total, total amount, net amount paid, settled amount, or equivalent final charge.
- Do not use subtotal if a final total exists.
- Do not use tax-only, tip-only, discount-only, or line-item-only amounts as expense_detail.amount unless the receipt clearly shows that only that item is being claimed.
- If both original currency and HKD-converted card settlement amounts appear, preserve the original transaction currency and original amount if the receipt clearly shows them; do not switch to HKD unless the receipt itself is HKD-only.
- If multiple unrelated purchases appear and the claimable one cannot be isolated, set amount to null.

When multiple dates appear:
- Prefer the transaction date or receipt date.
- Ignore print timestamp or payment processing timestamp unless it is the only clear date.

When merchant and line-item details appear:
- Use the clearest description of the claimed item or service for expense_detail.expense_description.
- Prefer receipt wording over generic summaries.
- Keep it concise and factual.

[Field Categories]
A. Stable Fields
These fields may be predicted from stable history or business rules:
submitted_to.type
submitted_to.target
contact_number

expense_detail.book_rate
Use historical values only if the historical profile indicates they are reliable enough.

B. Receipt-Derived Fields
These fields should primarily come from receipt evidence:
- expense_detail.currency
- expense_detail.amount
- expense_detail.expense_description
- expense_detail.attach_receipt

C. Context-Dependent Fields
These fields may use receipt evidence plus current context:
- expense_detail.expense_group
- expense_detail.business_purpose
- expense_detail.equivalent_hkd_amount
- expense_detail.declare_if_no_receipt

[Field Filling Rules]
submitted_to.type:
Must be either "Department" or "Specific Endorser".
If the current context clearly names a specific endorser, use "Specific Endorser".
If no specific endorser is named, use "Department" as the safer routing default.

submitted_to.target:
If submitted_to.type is "Department", use the safest department target.
If the historical profile does not provide a reliable department target, use "CPEG Office".
If submitted_to.type is "Specific Endorser", target should match the named endorser unless context clearly says otherwise.
submitted_to.target confidence must not exceed submitted_to.type confidence.

contact_number:
If current context provides a contact number, use it.
Otherwise, use the historical contact number only if it is a stable default.
If no stable value exists, return null.

supervisor_advisor_faculty:
If current context names a supervisor, advisor, or faculty member, use it.
Otherwise, use a historical value only if the historical profile marks it as reliable.
If the historical pattern is weak, return null or use a low-confidence suggestion only.

expense_detail.currency:
Use only configured currencies.
Use the receipt currency if explicitly shown.
If the receipt shows both a symbol and currency code, prefer the code when clear.
If the symbol is ambiguous and the extraction does not clearly identify the currency, use the structured extraction if it resolved the ambiguity; otherwise return null.
Do not default to HKD merely because history prefers HKD.

expense_detail.amount:
Use a number only if the receipt explicitly provides a clear final claimed amount.
If the receipt amount is ambiguous, inconsistent, or missing, return null.
Do not use history to fill amount.

expense_detail.book_rate:
If currency is "HKD", book_rate must be 1.0.
If the receipt or current context explicitly provides a reliable exchange rate for a non-HKD currency, use it.
Otherwise return null for non-HKD currencies.
Do not guess exchange rates from history.

expense_detail.equivalent_hkd_amount:
If amount and book_rate are both available, calculate amount * book_rate and round to 2 decimal places.
If either amount or book_rate is missing, return null.

expense_detail.attach_receipt:
If OCR or vision extraction exists from an uploaded receipt image or PDF, this should normally be true.
Only use false if the current context explicitly says the receipt is missing, illegible, unavailable, or not issued.

expense_detail.declare_if_no_receipt:
If attach_receipt is true, this must be "".
If attach_receipt is false and the current context gives a reason, write a short reasonable declaration.
If attach_receipt is false but no reason can be inferred, return null.

expense_detail.expense_description:
Use the clearest receipt-backed description of the actual expense item or service.
Do not invent item names, routes, events, vendors, or product details that are not visible in the receipt or extraction.
If only merchant-level evidence exists, use a short factual description based on the merchant or charge type.
If the receipt content is too vague, return null.

expense_detail.expense_group:
Infer only when the receipt or current context maps clearly to one of the configured expense groups.
Examples:
taxi, MTR, bus, transport, travel, fare, coach -> Transportation
airfare, flight, boarding pass, air ticket -> Airfare
ferry, overseas train, rail pass, intercity train -> Ferry/Train (Overseas)
hotel, accommodation, lodging -> Hotel
software, subscription, cloud, license, app store purchase -> Software
component, sensor, hardware, cable, electronics parts -> Lab Consumables/Electronics Components
lunch, refreshment, catering, meal -> Meal
registration, conference, workshop, seminar, symposium, visa fee, admission fee, ticket fee -> Registration/Conference/Visa Fee
postage, courier, shipping, delivery -> Delivery/Courier/Postage
membership, annual dues, society fee -> Membership Fee
publication fee, submission fee, APC -> Publication/Submission Fee
patent filing, patent application, IP fee -> Patent Fee
book, journal, textbook, article purchase -> Books/Journals
computer, laptop, monitor, desktop, computer accessory -> Computer
phone, tablet, portable device, mobile device -> Mobile Phone/Portable Electronic Devices
equipment, apparatus, instrument, tool -> Equipment
furniture, fixture, desk, chair, cabinet -> Furniture & Fixture
rental, hire, lease, venue hire, equipment hire -> Rental Fee
consumables, stationery, office supplies, general supplies -> General Consumables
service charge, vendor service, processing service -> Service Fee
If there is no clear evidence, return null.
Do not use weak history to fill this field as if it came from the receipt.

expense_detail.business_purpose:
Use a value only when the receipt or current context provides enough academic or business purpose.
Business purpose usually cannot be extracted from a receipt alone.
Do not invent project names, conference names, meeting names, locations, organizers, or research purposes.
If insufficient context exists, return null.

[Missing Fields Rules]
Add a field to missing_fields when:
The field is required for a complete claim but cannot be reliably determined.
The receipt does not support the field strongly enough.
The field needs direct user confirmation before submission.
Do not add declare_if_no_receipt to missing_fields when attach_receipt is true and declare_if_no_receipt is correctly "".

For each missing field, required_user_input_reason must include a short reason.
Examples:
"expense_detail.amount": "The receipt does not show a single clear final amount."
"expense_detail.business_purpose": "The receipt does not explain the academic or business purpose."

[Confidence Scoring Rules]
Each confidence score must be a number from 0.0 to 1.0.
Use this scale:
0.90-1.00: Explicitly shown in structured receipt extraction or clearly visible in OCR text, or directly determined by a fixed rule.
0.70-0.89: Strongly supported by receipt wording or clear current-context evidence.
0.40-0.69: Moderate inference from receipt evidence or moderate historical pattern.
0.10-0.39: Weak suggestion or conservative default.
0.00: Cannot be reliably determined, and the field value is null, empty, or missing.

[Hard Confidence Limits]
If amount is not explicitly supported by receipt evidence, expense_detail.amount confidence must be 0.0.
If currency is not explicitly supported by receipt evidence, expense_detail.currency confidence must not exceed 0.40.
If business_purpose is null, confidence must be 0.0.
If expense_description is null, confidence must be 0.0.
If contact_number is null, confidence must be 0.0.
If expense_group comes only from weak historical frequency, confidence must not exceed 0.35.
If expense_group comes from clear receipt keywords, confidence may be 0.65-0.85 depending on clarity.
If supervisor_advisor_faculty comes only from weak global history, confidence must not exceed 0.50.
equivalent_hkd_amount confidence must not exceed the lower confidence of amount and book_rate.
submitted_to.target confidence must not exceed submitted_to.type confidence.
If a value comes from historical default rather than receipt or current context, confidence must not exceed 0.85 unless the historical profile marks it as highly reliable.
